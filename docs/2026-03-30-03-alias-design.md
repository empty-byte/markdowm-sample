# 第 3 项设计：别名（Alias）

## 1. 背景与目标

别名用于提升内链输入与解析命中率，让同一文档可被多个名称引用（简称、旧名、英文名、拼音名等）。

目标：

- `[[文档名]]` 在标题不匹配时，能通过别名命中目标 `docId`
- 改标题时不破坏历史链接（依赖 `docId`）
- 为后续标签、图谱提供更稳定的引用层

## 2. 前置阻塞（必须直说）

当前代码仍是单文档编辑模型，尚未形成“文档实体层”（如 `docsIndex/currentDocId/doc switch`）。

这意味着：

- 别名可以先设计，但落地实现前必须先补“文档索引 + 当前文档上下文”
- 否则别名只能停留在单文档假数据，无法支撑真实内链解析

## 3. 范围

### In Scope（本期）

- 每个文档支持 `aliases: string[]`
- 内链解析支持 `title` 与 `alias` 双通道匹配
- 别名增删改的冲突校验
- 别名变更后，内链状态自动重算（ambiguous/dangling/resolved）

### Out of Scope（本期不做）

- 自动别名推荐（NLP）
- 跨工作区别名共享
- 别名版本历史回滚

## 4. 数据结构

逻辑模型（非实现代码）：

### 4.1 文档元信息

- `docId: ULID`
- `title: string`
- `aliases: string[]`
- `updatedAt: number`

### 4.2 别名归一化键（用于匹配）

- `normalizedAlias`：去首尾空格、连续空白折叠、大小写标准化
- 建议维护索引：`normalizedAlias -> docId[]`

说明：索引值是数组，因为别名可能冲突（是否允许冲突见“待你拍板项”）。

### 4.3 与内链解析关系

解析顺序建议：

1. `linkMemory` 命中
2. 标题精确匹配
3. 别名精确匹配
4. 单命中 -> resolved
5. 多命中 -> ambiguous（弹选择器）
6. 零命中 -> dangling

## 5. 状态流与交互流

1. 用户在文档设置中新增别名
2. 别名归一化并写入 `docsIndex`
3. 更新 `aliasIndex`
4. 触发受影响文档的内链重算
5. UI 侧表现为：
   - 原 dangling 可能转 resolved
   - 原 ambiguous 候选集可能变化

删除别名流程相同，方向相反。

## 6. 边界场景与异常处理

1. 别名与标题同名：优先标题命中
2. 别名与别名冲突：进入 ambiguous，不自动猜测
3. 标题改名但别名保留旧名：历史链接继续可解析
4. 别名为空字符串或超长：拒绝写入（建议长度上限 64）
5. 中英文混输：按归一化规则匹配，不做拼音模糊匹配

## 7. 验收标准（可测试）

1. 给文档 A 增加别名 `X` 后，`[[X]]` 可解析到 A
2. 删除别名 `X` 后，原 `[[X]]` 链接按规则回退（ambiguous/dangling）
3. 两文档共享别名 `X` 时，`[[X]]` 显示 ambiguous 并可人工选择
4. 改标题不影响按 `docId` 建立的已解析链接
5. 多端协同下，一端改别名，另一端解析结果实时更新

## 8. 与后续 MySQL 对齐点

建议表：

- `documents(doc_id, title, updated_at, ...)`
- `document_aliases(alias_id, doc_id, alias_text, normalized_alias, updated_at)`

索引建议：

- `idx_alias_normalized(normalized_alias)`
- `idx_alias_doc(doc_id)`
