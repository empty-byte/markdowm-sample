# Markdown 文档功能可行性报告

## 1. 报告背景

本项目当前定位为“Vue3 + Milkdown + Yjs + Hocuspocus”的在线文档编辑器 Demo，目标是验证类飞书文档场景下的核心能力可行性，包括：

- 可视化 Markdown 文档编辑
- 评论与锚点定位
- 手动历史快照与版本还原
- 多人协同编辑
- Markdown 面板按需查看与回写

本报告基于当前仓库实现情况，评估“Markdown 文档功能”继续推进为稳定产品能力的可行性、风险和建议实施路径。

## 2. 结论摘要

结论：可行性高，适合继续迭代。

原因：

- 编辑器核心方案已经收敛到单一 Milkdown 实现，技术路线清晰。
- Markdown 编辑、可视化编辑、评论、历史记录和多人协同已在同一页面内跑通。
- 当前项目已经具备 Demo 验证所需的大部分关键链路，后续更多是工程化补齐，而不是从零验证。

但需要明确：

- 当前更适合作为“功能验证版 / 内部演示版”。
- 若要进入稳定对外使用，还需要补齐持久化、鉴权、房间隔离、异常恢复、端到端测试等能力。

## 3. 当前项目基础能力评估

### 3.1 已实现能力

基于当前仓库，Markdown 文档功能已经具备以下基础：

- Milkdown 可视化编辑器已可稳定编辑 Markdown 内容。
- 下方 Markdown 面板支持显隐切换、手动应用和自动同步。
- 支持 slash 菜单、浮动工具栏、表格、图片、代码块、Latex 等增强能力。
- 支持正文选区评论、评论锚点高亮、点击正文定位右侧评论。
- 支持手动创建历史快照，并可还原对应正文与评论状态。
- 支持多人协同：同一房间内共享正文、评论和历史快照。
- 支持在线协作者 presence 展示。

从功能覆盖上看，项目已经不是“纯 Markdown 输入框”，而是一套带结构化交互的在线 Markdown 文档原型。

### 3.2 当前技术实现特点

- 前端框架：Vue 3 + TypeScript + Vite。
- 编辑器内核：Milkdown Crepe。
- 协同底座：Yjs + Hocuspocus。
- 当前协同服务：本地 Hocuspocus WebSocket 服务，默认地址为 `ws://127.0.0.1:1234`。
- 评论与历史快照已经不再只是本地状态，而是挂载到共享 Y.Doc 中同步。

这意味着当前方案不是临时拼接，而是围绕一个统一协同模型展开，后续延展性较好。

## 4. 可行性分析

### 4.1 技术可行性

评估：高。

理由：

- Milkdown 本身就是 Markdown-first 编辑器，适合承载“文档即 Markdown”的产品定位。
- 当前仓库已经同时具备“可视化视图”和“Markdown 原文视图”，并且支持互相同步。
- Yjs + Hocuspocus 已经跑通协同主链路，不需要再替换底层协议。
- 评论、历史和正文共用同一协同上下文，证明当前架构能承载文档周边能力，而不只是正文输入。

限制：

- 当前协同服务是内存态，重启后文档内容不会天然持久保留。
- 当前房间名是固定配置，更适合 Demo，不适合多文档生产环境。
- 当前仍是手动连接模式，交互上适合调试，但不适合正式产品默认体验。

### 4.2 产品可行性

评估：高。

理由：

- 现有功能已覆盖在线文档产品的最小可用集合：编辑、评论、历史、协同。
- 从交互上已经接近飞书 / Notion 类“可视化文档优先”的认知模型。
- Markdown 面板默认隐藏，有利于兼顾普通用户与技术用户。

适用场景：

- 内部知识文档
- 技术方案评审稿
- 团队协作记录
- 轻量级项目文档编辑器

当前不适合直接覆盖的场景：

- 强权限、多租户、复杂审批流的企业文档平台
- 需要稳定持久化审计和长期版本治理的正式生产文档系统

### 4.3 工程可行性

评估：中高。

理由：

- 现有代码已经完成单方案收敛，后续维护成本明显低于多编辑器并存方案。
- 评论、历史、命令和 slash 触发已经拆到 `src/features/editor-enhance/`，具备一定模块化基础。
- 有基本测试和构建验证链路。

当前工程短板：

- 测试仍以 helper 和基础行为为主，缺少多人协同端到端测试。
- 当前没有后端持久化层，也没有文档数据模型的服务端存储设计。
- 运行时错误展示较弱，调试主要依赖浏览器控制台和手工排查。

## 5. 风险与问题清单

### 5.1 高优先级风险

- 数据持久化缺失：当前协同服务脚本未接入数据库或文件存储，服务重启后内容可靠性不足。
- 文档房间固定：当前房间是固定值，不支持按文档 ID 动态隔离。
- 缺少权限控制：未实现登录、鉴权、访问控制和编辑权限划分。
- 历史记录为手动快照：更适合 Demo 和轻量版本管理，不等同于完整版本系统。

### 5.2 中优先级风险

- 协同连接目前为手动触发，用户容易误以为“未同步就是功能异常”。
- 当前评论与快照是共享状态，但缺少操作人、来源、变更原因等审计信息。
- 构建体积偏大，后续如果继续增加编辑器扩展，首屏成本可能继续上升。

### 5.3 低优先级风险

- 当前协作者昵称是本地生成，不支持用户自定义。
- Markdown 面板和编辑器之间的交互已经可用，但还缺少更强的冲突提示和变更反馈。

## 6. 继续落地所需补齐项

### 6.1 若目标是“稳定 Demo / 内部试用”

建议补齐：

- 页面加载后自动连接协同，减少手动操作成本。
- 增加连接错误、断线重连、同步失败的界面提示。
- 增加协作者昵称编辑能力。
- 增加基于浏览器或服务端的轻量持久化方案。

### 6.2 若目标是“正式产品化”

建议补齐：

- 服务端持久化：至少要保存正文、评论、历史快照和文档元数据。
- 动态文档路由与房间隔离：按文档 ID 创建协同房间。
- 用户体系：登录、鉴权、权限控制、协作者身份映射。
- 审计能力：记录谁在什么时候创建了评论、快照、正文修改。
- 端到端测试：双窗口协同、断线重连、历史恢复一致性、评论同步一致性。

## 7. 建议实施路径

### 阶段一：稳定现有 Demo

目标：把当前功能从“可演示”提升到“可稳定体验”。

建议内容：

- 自动连接协同
- 增加连接状态和错误提示
- 增加昵称设置
- 增加基础持久化
- 补双窗口联调清单

### 阶段二：支持多文档

目标：从单文档 Demo 进入“文档系统”雏形。

建议内容：

- 文档 ID 路由化
- 协同房间动态化
- 文档列表页与基础元数据
- 文档创建、重命名、删除

### 阶段三：产品化增强

目标：支持更真实的团队使用。

建议内容：

- 登录与权限体系
- 持久化版本体系
- 评论通知和状态流转
- 更完善的协作者光标 / 选区可视化
- 性能优化与监控

## 8. 最终判断

从当前项目状态看，Markdown 文档功能不仅“有实现可行性”，而且已经完成了大部分核心链路验证。

如果目标是：

- 做内部演示或 PoC：当前方案已经足够可行。
- 做小规模内部使用：补齐自动连接、持久化和基础错误提示后可推进试用。
- 做正式生产产品：仍需补齐服务端存储、权限、文档路由、审计和测试体系。

综合判断：

- 短期可行性：高
- 中期演进可行性：高
- 直接生产可用性：中

建议结论：继续沿当前技术路线迭代，而不是推倒重做或替换编辑器方案。
